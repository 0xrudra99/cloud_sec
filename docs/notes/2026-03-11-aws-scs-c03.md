## 1. IAM Policy Evaluation Chain

The mnemonic **D.S.B.I.R — "Don't Skip Boundaries In Reality"** captures the five evaluation gates every API request must pass:

| Letter | Gate | What it does |
|--------|------|-------------|
| D | Explicit Deny | Any explicit Deny in ANY policy kills the request. No exceptions. |
| S | SCP | Org-level fence. Can only restrict, never grant. |
| B | Permission Boundary | Ceiling. Effective perms = Identity Policy ∩ Boundary. |
| I | Identity Policy | Your actual grants. |
| R | Resource Policy | The lock on the resource. |

**Key rules:**
- Explicit Deny is absolute — overrides everything including root
- Permission boundary = intersection (ceiling), NOT union. Action must be in BOTH identity policy AND boundary
- Same-account: identity policy OR resource policy can grant (OR logic)
- Cross-account: BOTH identity policy AND resource policy must grant (AND logic)
- SCPs cannot grant permissions — they only restrict what accounts can do

> [!tip] Key Insight
> Think of the gates as concentric rings. A request must reach the center (ALLOW) by passing every ring. Any ring can stop it cold.

> [!warning] Common Misconception
> Permission boundaries do not add permissions — they set the maximum. If `s3:DeleteBucket` is not in the boundary, `s3:*` in the identity policy cannot grant it.

## 2. Permission Boundaries & Delegated IAM

A **permission boundary** is an IAM managed policy attached to a user or role that defines the maximum permissions that entity can ever have.

**The delegated IAM pattern** — safe way to let developers create IAM roles without privilege escalation risk:

```json
{
  "Effect": "Allow",
  "Action": ["iam:CreateRole", "iam:AttachRolePolicy"],
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "iam:PermissionsBoundary": "arn:aws:iam::123456789012:policy/ApprovedBoundary"
    }
  }
}
```

Plus a deny to prevent boundary removal:
```json
{
  "Effect": "Deny",
  "Action": "iam:DeleteRolePermissionsBoundary",
  "Resource": "*"
}
```

> [!tip] Key Insight
> The `iam:PermissionsBoundary` condition key forces developers to attach a specific boundary ARN when creating roles. Without this, a developer with `iam:CreateRole` could create a role with AdministratorAccess and assume it — a classic privilege escalation.

## 3. IAM Roles & Trust Policies

**IAM users** = permanent identity with long-term credentials (access key + password). No trust policy.

**IAM roles** = temporary identity that any entity can *assume* if listed in the trust policy.

| | IAM User | IAM Role |
|---|---|---|
| Who uses it | One specific human, permanently | Anyone in trust policy, temporarily |
| Credentials | Long-term | Short-term (expires 1-15 hrs) |
| Has trust policy? | No | Yes |
| Has permissions policy? | Yes | Yes |

Every role has **two separate policy documents**:
1. **Permissions policy** — what the role can DO
2. **Trust policy** — who can BECOME the role (resource-based policy on the role itself)

### Assuming a Role

"Assuming a role" = temporarily borrowing a different identity via `sts:AssumeRole`. Two checks must both pass:

1. **Trust policy on the role**: Is this entity listed as an allowed Principal?
2. **Identity policy on the caller**: Does the caller have `sts:AssumeRole` permission on that role's ARN?

Exception: AWS services (like EC2) don't need check 2 — AWS handles the credential exchange automatically.

### Who Can Be a Principal in a Trust Policy

| Principal type | Example |
|---|---|
| AWS service | `ec2.amazonaws.com` |
| IAM user (same account) | `arn:aws:iam::111:user/alice` |
| Entire account (cross-account) | `arn:aws:iam::999:root` |
| Federated identity | SAML, OIDC, Cognito |

### Cross-Account Role Assumption

**In Account B (role's account) — Trust policy:**
```json
{
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::111111111111:user/dev-bob" },
  "Action": "sts:AssumeRole"
}
```

**In Account A (caller's account) — Identity policy on dev-bob:**
```json
{
  "Effect": "Allow",
  "Action": "sts:AssumeRole",
  "Resource": "arn:aws:iam::222222222222:role/DataRole"
}
```

> [!tip] Key Insight
> If trust policy grants access to the entire Account A (`arn:aws:iam::111:root`), Account A's admin controls who gets access purely through identity policies — no need to modify the trust policy for each new user. Standard pattern for cross-account access at scale.

## 4. STS Tokens & Session Revocation

When `sts:AssumeRole` succeeds, STS returns **3 temporary credentials**:
1. `AccessKeyId`
2. `SecretAccessKey`
3. `SessionToken` — must be included with every API call

Default duration: **1 hour**. Max: **12 hours** (up to 36 hours with `--max-session-duration`).

### Why You Can't Delete STS Tokens

STS tokens are **stateless** — AWS does not store a list of active sessions. There is nothing to delete.

### How to Revoke Active Sessions

Add a deny policy to the role using the `aws:TokenIssueTime` condition:

```json
{
  "Effect": "Deny",
  "Action": "*",
  "Resource": "*",
  "Condition": {
    "DateLessThan": {
      "aws:TokenIssueTime": "2026-03-11T18:00:00Z"
    }
  }
}
```

This invalidates all tokens issued before the specified timestamp. New tokens issued after that time work normally.

In the AWS console, the **"Revoke active sessions"** button on a role does exactly this automatically.

> [!tip] Key Insight
> The condition key `aws:TokenIssueTime` is the only mechanism to invalidate in-flight STS sessions. This is the correct answer to any exam scenario involving compromised role credentials.

## 5. STS API Calls — Federation

All four STS calls for role assumption:

| API Call | When to Use |
|---|---|
| `AssumeRole` | Cross-account or same-account role switching |
| `AssumeRoleWithSAML` | Corporate AD/ADFS/Okta federation (SAML 2.0) |
| `AssumeRoleWithWebIdentity` | OIDC federation (Google, Facebook, Apple) — direct |
| `GetSessionToken` | MFA enforcement for IAM users |
| `GetFederationToken` | Custom identity broker (non-SAML/OIDC) |

### SAML 2.0 Federation Flow (Corporate AD → AWS Console)
1. Employee logs into company IdP (AD/ADFS/Okta)
2. IdP authenticates against Active Directory
3. IdP generates signed SAML assertion (XML token)
4. Browser sends SAML assertion to AWS
5. AWS calls `sts:AssumeRoleWithSAML`
6. STS returns temporary credentials
7. User accesses AWS Console

**AWS-side requirements:**
- SAML Identity Provider resource created in IAM (trusts IdP certificate)
- IAM roles with trust policy: `"Principal": {"Federated": "arn:aws:iam::111:saml-provider/MyAD"}`
- Role-to-group mapping configured in IdP

## 6. Cognito — User Pools vs Identity Pools

| | User Pools | Identity Pools |
|---|---|---|
| Purpose | Authentication — sign up/sign in | Authorization — temp AWS credentials |
| What it returns | JWT tokens | AWS access key + secret + session token |
| Use case | "Login with Google" for your app | App users need to call AWS services |
| Calls STS? | No | Yes — `AssumeRoleWithWebIdentity` under the hood |

**Key distinction:**
- **User Pools** = user directory + authentication layer
- **Identity Pools** = bridges authenticated users to AWS credentials

**Why Cognito Identity Pools over raw `AssumeRoleWithWebIdentity`:**
1. Multiple IdPs in one pool (Google + Facebook + SAML)
2. Unauthenticated (guest) access
3. Role mapping rules based on user attributes
4. App complexity hidden from developers

### Per-User S3 Isolation with Policy Variables

```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::game-saves/${cognito-identity.amazonaws.com:sub}/*"
}
```

`${cognito-identity.amazonaws.com:sub}` is a **policy variable** — replaced at evaluation time with the user's Cognito Identity ID. This is **ABAC in practice** — per-user S3 isolation at scale without separate policies per user.

> [!tip] Key Insight
> For mobile apps needing AWS credentials: use **Cognito Identity Pools** (not raw AssumeRoleWithWebIdentity). For corporate AD/SSO to AWS Console: use **SAML federation** with `AssumeRoleWithSAML`.

## 7. Confused Deputy Problem & External ID

**The problem:** A SaaS service has permission to assume roles across many customer accounts. A malicious customer tricks the SaaS into assuming a different customer's role — using the SaaS as an unwitting middleman.

**The fix — External ID:**
- Customer generates (or SaaS assigns) a unique secret ID per customer relationship
- Customer embeds it as a required condition in their role's trust policy
- SaaS must supply it on every `AssumeRole` call
- Malicious customer doesn't know victim's External ID → assumption fails

```json
{
  "Effect": "Allow",
  "Principal": {"AWS": "arn:aws:iam::SAAS_ACCOUNT:root"},
  "Action": "sts:AssumeRole",
  "Condition": {
    "StringEquals": {"sts:ExternalId": "acct-a-f7k2p9"}
    }
}
```

> [!tip] Key Insight
> External ID = secret handshake between customer and SaaS. Unique per customer-SaaS relationship. Exam pattern: any third-party vendor/SaaS assuming roles in your account → External ID prevents confused deputy.

## 8. Role Chaining & Service-Linked Roles

### Role Chaining
Assuming a role from within an already-assumed role session.

**Hard rule:** Role chaining caps session duration at **1 hour — always**, regardless of the role's `--max-session-duration` setting.

| Method | Max Duration |
|---|---|
| Direct AssumeRole from IAM user | Up to 12 hours |
| Direct AssumeRole from AWS service | Up to 12 hours |
| Role chaining (role → role) | **1 hour hard cap** |

> [!warning] Common Exam Trap
> App uses Lambda → assumes Role A → assumes Role B. Sessions expire after 1 hour and app breaks. Fix: have Lambda assume Role B directly, eliminating the chain.

### Service-Linked Roles
Pre-defined roles created and managed by AWS for specific services.

| | Regular Role | Service-Linked Role |
|---|---|---|
| Who defines permissions | You | AWS |
| Can you modify permissions? | Yes | No |
| Can you delete it? | Yes | Only when service stops using it |
| Created by | You | Automatically when service is enabled |

**Security benefit:** AWS enforces least privilege — service can never be accidentally over-permissioned.
**Exam pattern:** "Ensure AWS service X only has minimum permissions" → service-linked roles.

## 9. IAM Identity Center (AWS SSO)

Centralized identity management for multi-account AWS Organizations. One login → access to any permitted account.

**Core components:**

| Component | What it is |
|---|---|
| Permission Set | Reusable access template (e.g., ReadOnly, Developer, Admin) |
| Assignment | 3-way link: User/Group + Permission Set + Account |
| Credential vending | Auto-issues temporary STS credentials when user clicks into an account |

**How it works:**
1. Employee logs into Identity Center portal once
2. Sees list of accounts they're assigned to
3. Clicks an account → Identity Center calls STS → temporary credentials dispensed
4. Credentials expire automatically

**Identity Center auto-creates IAM roles** in every assigned account after you configure assignments. No manual role creation across accounts.

**SCIM — automated user sync:**
- Connects Identity Center to external IdP (Okta, Azure AD)
- Automatically syncs users/groups as people join or leave
- Employee removed from Okta → SCIM removes from Identity Center → AWS access revoked immediately
- **Exam pattern:** "Ensure terminated employees lose AWS access automatically" → SCIM sync

**When to use what:**

| Scenario | Solution |
|---|---|
| Corporate employees, multi-account access | IAM Identity Center |
| Corporate employees, single account Console | SAML federation directly |
| Mobile/web app users needing AWS creds | Cognito Identity Pools |
| On-premises servers needing AWS API access | IAM Roles Anywhere |

> [!tip] Key Insight
> SCPs and Identity Center solve different problems. SCPs restrict what accounts CAN do. Identity Center controls WHO can access accounts. You need both.

## 10. ABAC vs RBAC

**RBAC (Role-Based Access Control):** Access based on job function/role. Someone manually assigns a role to a user.
> "Alice gets the Developer role" — a human made a decision

**ABAC (Attribute-Based Access Control):** Access based on tags/attributes on both user AND resource. Policy matches them automatically.
> "Whoever has `team=payments` gets payments resources" — tags make the decision

```json
{
  "Effect": "Allow",
  "Action": "s3:*",
  "Resource": "*",
  "Condition": {
    "StringEquals": {
      "aws:ResourceTag/Team": "${aws:PrincipalTag/Team}"
    }
  }
}
```

One policy. Works for all teams, all resources, forever — as long as tags are consistent.

| Use RBAC when | Use ABAC when |
|---|---|
| Small number of roles | Many teams/environments/resources |
| Stable access patterns | Frequently changing access patterns |
| Simple environments | Fine-grained dynamic access needed |

> [!tip] Key Insight
> The Cognito S3 prefix scenario (`${cognito-identity.amazonaws.com:sub}`) is ABAC — a user attribute automatically determines which resource they can access. No manual per-user assignment.

## 11. IAM Access Analyzer

Three capabilities:

| Feature | What it does |
|---|---|
| External access findings | Flags resources accessible outside account/org boundary |
| Policy validation | Checks policies for errors and security issues before deploying |
| Unused access findings | Shows permissions granted but never used — right-size to least privilege |

**Resource types scanned:** S3 buckets, IAM roles, KMS keys, Lambda functions, SQS queues, SNS topics, Secrets Manager secrets, EFS file systems.

**Zone of trust:** Define as your account OR your entire Organization. Anything outside = finding.

**Investigation pattern for unexpected permissions:**
1. CloudTrail — find exact API calls, credentials used, timestamps
2. IAM Policy Simulator — test user's current policies against the action
3. IAM Access Analyzer — check resource-based policy on the target resource
4. Check for role assumption in CloudTrail — user may have assumed a permissive role

> [!warning] Common Misconception
> "User can do something their identity policy doesn't allow" → check the resource-based policy. Same-account OR logic means the bucket/resource policy alone can grant access, bypassing identity policy restrictions.

## 12. IAM Fundamentals

**IAM Users — credentials:**
- Console password — for AWS Management Console login
- Access keys (AccessKeyId + SecretAccessKey) — for CLI/API access
- Maximum **2 access keys** per user — allows rotation without downtime
- Access keys never expire — must be manually disabled/deleted

**Password policy (account-level):** minimum length, character requirements, expiration, reuse prevention.

**IAM Groups:**
- Containers for users only — no nesting (no groups within groups)
- **Cannot be a Principal** in any policy — this is a common exam trap
- Users inherit all policies attached to their groups
- A user can belong to multiple groups — permissions are unioned
- No default group — new users start with no permissions

**IAM Policies — three types:**

| Type | Creator | Reusable? | Exam use case |
|---|---|---|---|
| AWS managed | AWS | Yes | Standard permissions (ReadOnlyAccess, AdministratorAccess) |
| Customer managed | You | Yes | Custom org permissions — preferred |
| Inline | You | No — embedded in one entity | Strict 1-to-1 binding, deleted with entity |

**PARC model** — every policy statement has:
- **P**rincipal — WHO (only in resource-based policies, not identity-based)
- **A**ction — WHAT
- **R**esource — WHICH resource
- **C**ondition — UNDER what circumstances

## 13. Remaining Policy Types

### SCPs — Full Coverage

**Two strategies:**
- **Deny list (default):** AWS attaches `FullAWSAccess` SCP to all accounts. Add Deny SCPs on top. Simpler.
- **Allow list:** Remove `FullAWSAccess`. Add only explicit Allows. More restrictive for high-security environments.

**SCP inheritance:** Cumulative top-down — Root → OU → Sub-OU → Account. Any Deny in the chain blocks the action.

**Three things SCPs cannot do:**
1. Grant permissions
2. Affect the management account
3. Affect service-linked roles

### Resource Control Policies (RCPs) — NEW in C03

Resource-side equivalent of SCPs. Controls what can be done TO resources (vs SCPs which control what principals can do).

**Key use case:** Prevent data exfiltration — deny S3 access from principals outside your org:
```json
{"Condition": {"StringNotEquals": {"aws:PrincipalOrgID": "o-xxxxxxxxxxxx"}}}
```

| | SCP | RCP |
|---|---|---|
| Controls | What principals CAN do | What can be done TO resources |
| Direction | Principal-centric | Resource-centric |

### S3 ACLs
Legacy mechanism. Grant READ/WRITE/FULL_CONTROL on buckets/objects to other accounts or predefined groups (AllUsers = public). Best practice: disable ACLs, use bucket policies instead. Block Public Access overrides ACLs.

### Session Policies
Passed inline when creating STS sessions. Further restrict the session beyond the role's permissions. Effective permissions = role permissions ∩ session policy. Use case: CI/CD pipeline scoping individual job sessions.

## 14. Policy Evaluation — Deep

### Cross-Account Deep
For cross-account resource access (e.g., KMS key in Account B accessed by Account A user):
- Resource policy in Account B must explicitly allow the external principal
- Identity policy in Account A must allow the action on that resource ARN
- Both required — no exceptions

### VPC Endpoint Policies
Additional filter layer: User → Identity Policy → VPC Endpoint Policy → Resource Policy → ALLOW/DENY. All must allow. Common exam pattern: restrict S3 endpoint to only allow access to company buckets — prevents exfiltration to personal S3 accounts.

### Implicit vs Explicit Deny

| | Implicit Deny | Explicit Deny |
|---|---|---|
| What it is | Default — no Allow exists | A `"Effect": "Deny"` statement |
| Override? | Yes — Allow in any policy overrides | No — nothing overrides |
| With resource policies | Identity policy can still grant (same-account) | Cannot be overridden by any policy |

## 15. Roles — Deep Mechanics

### Instance Profiles
Wrapper that holds exactly ONE IAM role for EC2. Created automatically in console, manually in CLI/IaC. EC2 retrieves temp credentials from IMDS at `169.254.169.254`.

### Execution Roles

**Lambda:** Execution role = what Lambda can call. Resource policy = who can invoke Lambda. These are separate.

**ECS — two separate roles (exam trap!):**
- Task execution role — ECS agent pulls images, writes logs (infrastructure)
- Task role — application code inside container (business logic)

**EKS IRSA (IAM Roles for Service Accounts):** Individual pods assume IAM roles via OIDC projected tokens. Fine-grained per-pod permissions — not all pods sharing one role.

### IAM Roles Anywhere — NEW in C03
On-premises servers get temp AWS credentials using X.509 certificates from your PKI/ACM PCA. No hardcoded long-term keys. Exam pattern: "on-prem servers need AWS API access without storing credentials" → Roles Anywhere.

## 18. IAM Best Practices

### Root User
- Never use for daily tasks
- Enable MFA immediately
- No access keys for root
- CloudTrail alert on any root usage
- Break-glass: credentials in physical safe
- NEW in C03: centralized root access management for member accounts via Organizations

### Access Key Rotation
2-key rotation pattern: create new → update apps → verify → deactivate old → delete old. Target: 90 days. Better: replace with IAM roles entirely.

### MFA Enforcement
Use `BoolIfExists` (not `Bool`) for MFA conditions — handles service calls that don't support MFA:
```json
{"Condition": {"BoolIfExists": {"aws:MultiFactorAuthPresent": "false"}}}
```

### Presigned URLs vs Signed URLs

| | S3 Presigned URL | CloudFront Signed URL | CloudFront Signed Cookie |
|---|---|---|---|
| Accesses | S3 directly | CloudFront (cached) | CloudFront (cached) |
| IP restriction | No | Yes | Yes |
| Covers | Single object | Single object | Multiple objects |
| Use when | Quick temp S3 access | IP restriction + CDN | Multiple files (e.g., streaming) |

### Separation of Duties
Separate roles for approval and deployment. No single person can do both. Production access only via CI/CD pipelines.

### Delegated Administrator Accounts
For org-wide services (GuardDuty, Security Hub, Macie, Inspector, Access Analyzer): designate a dedicated security account as delegated admin. Management account used only for billing/org management.

### Amazon Verified Permissions — NEW in C03
Fine-grained application-level authorization using Cedar policy language. Use when IAM is too coarse — e.g., row-level access control, "can Alice view document 123?"

### AWS Directory Service

| Service | What it is | Use when |
|---|---|---|
| Managed Microsoft AD | Full AD in AWS | Want AD in AWS, or need both AWS + on-prem AD |
| AD Connector | Proxy to on-prem AD | Keep existing on-prem AD, just connect AWS to it |
| Simple AD | Lightweight LDAP | Small org, basic LDAP, no full AD features needed |

## 19. KMS — Complete Coverage

### Envelope Encryption (the core concept)
KMS never encrypts your data directly (4KB limit). Instead:

1. S3 calls `kms:GenerateDataKey` → KMS returns: **plaintext data key** + **encrypted data key**
2. S3 encrypts your file with the plaintext data key (locally, in memory)
3. S3 discards the plaintext data key
4. S3 stores: encrypted file + encrypted data key together

On download:
1. S3 sends encrypted data key to KMS → `kms:Decrypt`
2. KMS decrypts it → returns plaintext data key
3. S3 decrypts the file → you download it

> [!tip] Key Insight
> The KMS key (CMK) never leaves KMS and never touches your file directly. It only ever encrypts/decrypts the small data key. The data key does the actual file encryption locally.

### KMS Key Types

| Type | Managed by | Audit trail | Key policy control | Cost |
|---|---|---|---|---|
| AWS owned | AWS | None — invisible | None | Free |
| AWS managed | AWS | CloudTrail | None (can't modify) | Free |
| Customer managed (CMK) | You | CloudTrail | Full control | $1/month + API |

Use CMK when: auditor needs proof of control, cross-account access, custom rotation, disable/delete capability.

### Key Policies — The Primary Control

**IAM policies alone cannot grant KMS access.** The key policy must explicitly allow it. KMS is the exception to the standard IAM rule.

Two patterns:
- **Root grant** (`arn:aws:iam::111:root`): defers to IAM — IAM policies can now grant access
- **Specific principal only**: only those principals, IAM policies irrelevant

Key policy alone (without IAM policy) CAN grant access for same-account principals.
For cross-account: both key policy AND IAM policy in the other account required.

> [!warning] Exam Trap
> Key policy has root grant + Lambda execution role has kms:Decrypt = **ALLOWED**. The root grant means IAM is in charge — Lambda's IAM policy is sufficient.

### KMS Grants
Programmatic temporary delegation of KMS access. Created via `kms:CreateGrant`, retired via `kms:RetireGrant`. Used when applications need to dynamically delegate key access without modifying key policies.

### Encryption Context
Optional key-value pairs passed during encrypt/decrypt:
- Logged in CloudTrail with every operation
- Must provide exact same context to decrypt — wrong context = failure
- Not stored with ciphertext — you must track it
- Binds ciphertext to its intended purpose

### Key Rotation
- **Automatic**: Annual, generates new key material, old material retained, same key ID/ARN, symmetric CMKs only
- **Manual**: Create new CMK, update alias to point to it, old key retained for decryption
- **Cannot auto-rotate**: asymmetric keys, imported key material

### Symmetric vs Asymmetric
- **Symmetric**: one key encrypts + decrypts, AWS service integration, auto-rotation supported
- **Asymmetric**: key pair, public key downloadable, digital signatures, external parties encrypting data

### Key States & Deletion
- Enabled → Disabled → Pending deletion (7-30 day wait) → Deleted
- Data encrypted with deleted key = **permanently unrecoverable**
- Best practice: CloudWatch alarm on kms:Decrypt during pending deletion window
- Imported key material: can delete immediately, no waiting period

### Multi-Region Keys
Same key material across regions. Same key ID suffix. Encrypt in us-east-1, decrypt in eu-west-1 without re-encrypting. Use for: global DynamoDB, active-active apps, DR.

### Cross-Account Key Sharing
Key policy in Account A must grant Account B's principal + IAM policy in Account B must allow the action. Both required (AND logic).

### KMS vs CloudHSM

| | KMS | CloudHSM |
|---|---|---|
| Hardware | Multi-tenant | Single-tenant (dedicated) |
| FIPS | Level 2 | **Level 3** |
| Key ownership | AWS manages infra | **You own keys — AWS cannot access** |
| AWS service integration | 100+ services | Via custom key store only |
| Cost | $1/key/month | ~$1.60/hour per HSM |
| Use when | Standard encryption | Regulatory, Oracle TDE, SSL offload, code signing |

**Custom key store**: KMS + CloudHSM = AWS service integration + Level 3 FIPS + dedicated hardware.

## 20. Secrets Manager & Parameter Store

### Secrets Manager
Stores + auto-rotates secrets. Rotation uses Lambda (AWS-provided for RDS/Aurora/Redshift/DocumentDB, custom for others). Secret versions: AWSCURRENT, AWSPREVIOUS, AWSPENDING. Multi-region replication. Resource-based policies.

### Parameter Store
General config + secrets. SecureString = KMS-encrypted. Hierarchical paths (`/app/prod/db`). No auto-rotation. Free standard tier. Max 4KB (8KB advanced).

| | Parameter Store | Secrets Manager |
|---|---|---|
| Auto-rotation | No | Yes |
| Cross-region | No | Yes |
| Cost | Free | $0.40/secret/month |
| Use when | Config + simple secrets | DB creds needing rotation |

## 21. ACM & ACM Private CA

**ACM:** Public SSL/TLS certs, auto-renewal, integrates with CloudFront/ALB/NLB/API Gateway. Cannot export. Cannot use directly on EC2.

**ACM PCA:** Private CA hierarchy, exportable certs, CRL/OCSP revocation, ~$400/month per CA.

> [!warning] Exam Trap
> SSL on EC2 instance → cannot use ACM directly. Put ALB in front with ACM cert, or use third-party cert on EC2.

## 22. S3 Encryption — All Four Types

| Type | Key managed by | Audit trail | Header |
|---|---|---|---|
| SSE-S3 | S3/AWS | No | `AES256` |
| SSE-KMS | KMS | Yes (CloudTrail) | `aws:kms` |
| SSE-C | You (per request) | No | Key in request header |
| CSE | You (client-side) | No | Data encrypted before upload |

**Enforce SSE-KMS via bucket policy:**
```json

```
Effect: Deny — blocks any upload not using SSE-KMS.

**S3 bucket keys:** One data key per bucket instead of per object — reduces KMS API calls and cost.

## 23. S3 Object Lock & Data Protection

**S3 Object Lock — requires versioning enabled first (exam trap!)**

| Mode | Who can override? | Use case |
|---|---|---|
| Governance | Users with s3:BypassGovernanceRetention | Internal compliance |
| Compliance | Nobody — not even root or AWS | Regulatory (SEC 17a-4, HIPAA) |

Retention period can be **extended but not shortened** in Compliance mode.
Legal hold: indefinite lock, no expiry, removed explicitly.

**Glacier Vault Lock:** WORM for Glacier. Lock vault policy — once locked, policy itself is immutable.

### Data Masking (NEW in C03)
- **CloudWatch Logs data protection:** Masks PII (SSNs, credit cards, AWS creds) in logs automatically. Shows as `***`. No code changes needed.
- **SNS message data protection:** Detects/masks PII in SNS messages before delivery.

### Compliance S3 Pattern (4 steps)
1. Enable versioning (prerequisite for Object Lock)
2. S3 Object Lock — Compliance mode + retention period
3. SSE-KMS with customer managed key
4. Bucket policy denying unencrypted uploads

## 24. EBS Encryption

**EBS (Elastic Block Store)** = network-attached block storage for EC2. Think of it as an external hard drive plugged into your EC2 instance over AWS's internal network. Persists independently of the EC2 instance — detach, terminate the instance, volume survives.

**Storage type comparison:**
| Service | Type | Key trait |
|---|---|---|
| EBS | Block | Attached to one EC2, persists independently |
| EFS | File (NFS) | Shared across multiple EC2 instances |
| S3 | Object | Files/backups, not mountable as filesystem |
| Instance Store | Block (physical) | Ultra-fast, LOST when EC2 stops |

### Encrypting an existing unencrypted volume (no in-place encryption)
```
1. Create snapshot of unencrypted volume
2. Copy snapshot → enable encryption → select CMK
3. Create new EBS volume from encrypted snapshot
4. Stop EC2 instance
5. Detach old unencrypted volume
6. Attach new encrypted volume
7. Start EC2 instance
```

### Encryption propagation rule
- Unencrypted volume → snapshot = unencrypted snapshot
- Copy snapshot WITH encryption = encrypted snapshot
- Any volume/AMI created from encrypted snapshot = automatically encrypted

### EBS Default Encryption (account-level)
Single toggle per account per region. All new volumes auto-encrypted. Zero developer action required.
```bash
aws ec2 enable-ebs-encryption-by-default
aws ec2 modify-ebs-default-kms-key-id --kms-key-id arn:aws:kms:...:key/your-cmk
```

### Defense-in-depth pattern
| Layer | Control |
|---|---|
| Preventive (primary) | EBS default encryption enabled |
| Preventive (backstop) | SCP denying `ec2:RunInstances` unless `ec2:Encrypted: true` |
| Detective | AWS Config rule `encrypted-volumes` |
| Responsive | Config auto-remediation → snapshot+encrypt+swap |

### Exam traps
- Encrypted snapshot using **AWS managed key** → cannot share with other accounts
- Encrypted snapshot using **CMK** → can share, but must also share the CMK key policy with target account
- Copying an AMI cross-region → must specify a KMS key in the **destination region** (keys are region-bound)
- EBS encryption uses **AES-256**, KMS manages the data key (unique per volume)

> [!tip] Key Insight
> "No unencrypted volumes, minimal friction" → EBS default encryption (one toggle). "Enforce across org, no exceptions" → SCP as backstop.

> [!warning] Common Misconception
> You cannot encrypt an existing EBS volume in place. There is no "flip a switch" option. The only path is: snapshot → copy with encryption → new volume → swap.

## 25. RDS/Aurora Encryption

**RDS** = managed relational database (MySQL, PostgreSQL, Oracle, SQL Server, Aurora). Storage is EBS underneath — same encryption constraints apply.

**Core concepts:**
- **DB Instance** — the running database server. Has an endpoint (DNS), engine, size, port. You connect via endpoint, never SSH in.
- **Read Replica** — async copy for read scaling. Can be promoted to primary for DR.
- **Multi-AZ** — sync standby in another AZ for automatic failover. Silent — no traffic until primary fails.
- **Parameter Group** — config file for DB engine. Create custom group to modify settings (e.g., force SSL).

**Multi-AZ vs Read Replica:**
| | Multi-AZ | Read Replica |
|---|---|---|
| Purpose | High availability | Read scaling |
| Replication | Synchronous | Asynchronous |
| Serves traffic? | No | Yes |
| Failover? | Automatic | Manual promotion |

### Encrypting an existing unencrypted RDS instance
```
1. Create DB snapshot
2. Copy snapshot → enable encryption → select CMK
3. Restore new DB instance from encrypted snapshot
4. Update app connection string
5. Delete old instance
```

### Encryption rules
- Cannot enable encryption on a running instance — must use snapshot+restore
- Encrypted primary → read replicas automatically encrypted
- Unencrypted primary → replicas CANNOT be encrypted independently
- Cross-region read replica → requires KMS key in destination region + automated backups enabled on source

### SSL/TLS in transit (parameter group)
- PostgreSQL: `rds.force_ssl = 1`
- MySQL/Aurora MySQL: `require_secure_transport = ON`
- SQL Server: "Force Encryption" in option group

### IAM database authentication
Replaces username/password with short-lived IAM auth token (15 min validity).
- Works for MySQL and PostgreSQL only
- Token obtained via STS, used as password
```json
{
  "Effect": "Allow",
  "Action": "rds-db:connect",
  "Resource": "arn:aws:rds-db:us-east-1:123456789012:dbuser:db-ABCDEFG/mydbuser"
}
```

> [!tip] Key Insight
> Cross-region read replica needs TWO things: (1) KMS key in destination region, (2) automated backups enabled on source instance.

> [!warning] Common Misconception
> Multi-AZ standby does NOT serve read traffic — it is invisible to the app until the primary fails. Only read replicas serve active traffic.

## 26. DynamoDB Encryption

DynamoDB = fully managed NoSQL (key-value + document). Encryption at rest is **always on — cannot be disabled**.

| Key type | Audit trail | Control |
|---|---|---|
| AWS owned (default) | No CloudTrail | None |
| AWS managed (`aws/dynamodb`) | Yes | Cannot modify key policy |
| CMK | Yes | Full control |

**The critical distinction:** DynamoDB has **no resource-based policy**. Cross-account access requires IAM role assumption — cannot be done via a resource policy like S3 or KMS.

> [!warning] Common Misconception
> "Compliance requires audit trail of DynamoDB key usage" → AWS owned key has NO CloudTrail visibility. Must switch to AWS managed key or CMK.

## 27. EFS Encryption

**EFS (Elastic File System)** = managed NFS share. Multiple EC2 instances can mount simultaneously (unlike EBS which is one instance at a time).

**Two independent encryption dimensions:**

| Dimension | How | Enforced by |
|---|---|---|
| At rest | KMS (AES-256), set at creation | Cannot enable on existing EFS |
| In transit | TLS via mount helper | `amazon-efs-utils` + `-o tls` flag |

```bash
# Insecure mount (data in transit unencrypted)
sudo mount -t nfs4 fs-abc123.efs.us-east-1.amazonaws.com:/ /mnt/efs

# Secure mount (TLS)
sudo mount -t efs -o tls fs-abc123:/ /mnt/efs
```

**Enforce TLS via EFS resource policy:**
```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "*",
  "Condition": { "Bool": { "aws:SecureTransport": "false" } }
}
```

> [!warning] Common Misconception
> At-rest encryption and in-transit encryption are independent. A system can be encrypted at rest but transmitting data in the clear if TLS mount isn't configured. Both must be set separately.

## 28. Data-in-Transit Encryption & VPC Endpoints

### VPC Endpoints — keeping traffic off the public internet

**Gateway Endpoint** (S3 + DynamoDB only, free):
- Adds a route table entry → traffic stays within AWS network
- No ENI created, no private IP

**Interface Endpoint / PrivateLink** (all other AWS services, has cost):
- Creates an ENI with private IP inside your VPC
- Private DNS replaces public endpoint automatically
- Accessible from on-premises via Direct Connect/VPN

| | Gateway | Interface |
|---|---|---|
| Services | S3, DynamoDB | Everything else |
| Cost | Free | Per-hour + data |
| On-prem access | No | Yes |

### Enforce VPC-only S3 access
```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"],
  "Condition": {
    "StringNotEquals": { "aws:SourceVpc": "vpc-12345678" }
  }
}
```

Use `aws:SourceVpc` not `aws:SourceIp` — VPC endpoint rewrites source IP, making SourceIp unreliable.

### PrivateLink
Expose your own service privately to other VPCs/accounts:
`Your Service → NLB → Interface Endpoint (ENI in consumer VPC)`
No VPC peering needed, no internet, consumer accesses via private IP.

### AWS Verified Access (NEW in C03)
Zero-trust network access — VPN replacement:
- Checks identity (IAM Identity Center / OIDC) + device posture (certificate, compliance)
- Both must pass before proxying request to application
- Application stays in private subnet, never exposed to internet

### MACsec
Layer 2 hardware encryption for **AWS Direct Connect** physical links.
Exam trigger: "Encrypt data on the physical Direct Connect connection" → MACsec.

### ELB Security Policies
- ALB/NLB terminate TLS — choose security policy controlling TLS versions and cipher suites
- Enforce TLS 1.2+ by selecting appropriate predefined security policy
- ACM certificates used on ALB (cannot use ACM certs directly on EC2)

### Redshift / OpenSearch / Kinesis (at rest)
All follow the same pattern: KMS at creation, cannot enable on existing resource.
- Redshift: 4-tier key hierarchy, CloudHSM supported for FIPS, `require_ssl = true` parameter
- OpenSearch: node-to-node encryption + HTTPS, fine-grained access control via IAM
- Kinesis Data Streams: SSE with KMS per stream

> [!tip] Key Insight
> SSL/TLS encrypts the data traveling — VPC Endpoints change the network path. Both are needed for full protection. SSL without VPC Endpoint = encrypted but still traverses public internet.

> [!warning] Common Misconception
> `aws:SourceIp` breaks when traffic goes through a VPC endpoint. Always use `aws:SourceVpc` or `aws:SourceVpce` for VPC endpoint-based access controls.

## 29. S3 Advanced Data Protection

**S3 Versioning + MFA Delete:**
- Versioning: delete creates a "delete marker" — object hidden, not gone. Restore by removing delete marker.
- MFA Delete: requires MFA token to permanently delete a version or disable versioning. Only root user can enable.

**S3 Block Public Access:**
- 4 settings at bucket or account level — override any bucket policy/ACL attempting public access
- Account-level = nuclear option: no bucket in account can ever be public
- Exam pattern: org-wide → Block Public Access at account level + SCP preventing it from being disabled

**S3 Access Points:**
- Named endpoints on a bucket, each with own policy — replaces one giant bucket policy for many teams
- Can be restricted to a specific VPC (VPC-only access point)

**S3 Access Logging:**
- Records every request (who, what, from where, when). Detective control, not real-time.
- Logs written to a **different** target bucket — never the same bucket (infinite loop).

**S3 CORS:**
- Allows browser JS from one domain to access S3 from another. Misconfigured wildcard origins = security risk.

**AWS Backup:**
- Centralized backup: EC2, EBS, RDS, DynamoDB, EFS, S3, FSx
- **Backup Vault Lock** — WORM on backup vault. Even root cannot delete locked backups. Ransomware protection.
- Cross-account backup — send to separate account (ransomware in primary can't reach backups)

**Amazon Macie:**
- ML-powered sensitive data discovery for S3: PII, PHI, PCI, AWS credentials, custom patterns
- Findings surface in Security Hub
- Exam trigger: "automatically identify S3 buckets with sensitive data" → Macie

**Client VPN:**
- Managed OpenVPN service. Remote users connect via TLS tunnel into VPC.
- Auth: certificate (mutual TLS), Active Directory, or SAML/SSO

**Nitro Inter-node Encryption (NEW in C03):**
- Modern EC2 on Nitro hypervisor: traffic between instances auto-encrypted at hardware level. Zero config, no perf penalty.
- EMR, EKS, SageMaker: configurable inter-node TLS encryption for distributed workloads
- Exam trigger: "encrypt EC2-to-EC2 traffic without performance impact" → Nitro instance types

## 30. VPC Fundamentals & VPC Endpoints (Deep Dive)

### VPC — Your Private Network in AWS
- VPC = isolated private network. Resources inside share a private IP range (CIDR block, e.g. `10.0.0.0/16`)
- **Subnet** = segment of VPC tied to one AZ (e.g. `10.0.1.0/24`)
- **Route table** = traffic signposts per subnet. `0.0.0.0/0 → IGW` = public subnet. `0.0.0.0/0 → NAT` = private subnet
- **Internet Gateway (IGW)** = bidirectional door to internet. EC2 needs public IP + IGW route
- **NAT Gateway** = one-way outbound-only door. Private EC2 reaches internet; internet cannot reach in. Lives in public subnet.

```
Public subnet:  route 0.0.0.0/0 → IGW    (bidirectional)
Private subnet: route 0.0.0.0/0 → NAT    (outbound only)
```

### The Problem VPC Endpoints Solve
Without endpoints, EC2 in a private subnet calling S3/KMS must go:
`EC2 → NAT Gateway → Internet → AWS service public IP`
Traffic leaves AWS network, incurs NAT costs, security risk.

### Gateway Endpoint (S3 + DynamoDB only)
- Free. No ENI. Just a route table entry pointing to AWS's internal network.
- AWS adds a prefix list route (`pl-xxxxx → vpce-xxxxx`) to your route table automatically
- Traffic: `EC2 → route table → Gateway Endpoint → S3/DynamoDB` (never touches internet)
- Cannot be used from on-premises

### Interface Endpoint / PrivateLink (all other services)
- Creates an **ENI** (virtual network card) with a **private IP inside your VPC subnet**
- ENI has one foot in your VPC (private IP) and one foot on AWS's internal backbone
- **Private DNS** overrides DNS resolution: `kms.amazonaws.com` → resolves to ENI's private IP (e.g. `10.0.1.45`) instead of public IP
- Your app code doesn't change. Route table routes `10.0.1.45` locally. Never hits internet.
- Cost: ~$0.01/hr per AZ + $0.01/GB
- **Accessible from on-premises** via Direct Connect/VPN (because it's a real private IP)

```
WITHOUT endpoint:
EC2 → NAT → Internet → KMS (public IP 54.x.x.x)

WITH Interface Endpoint:
EC2 → ENI (10.0.1.45, in your VPC) → AWS backbone → KMS
      DNS answered 10.0.1.45 instead of 54.x.x.x — stays local
```

### Endpoint Policies
Additional access control layer on the endpoint itself:
```json
{
  "Effect": "Allow", "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-company-bucket/*"
}
```
Prevents data exfiltration — compromised EC2 cannot reach attacker's S3 bucket through your endpoint.

### Decision Matrix
| Need | Solution |
|---|---|
| S3/DynamoDB private access | Gateway Endpoint (free) |
| KMS/STS/SSM private access | Interface Endpoint |
| On-premises → AWS service privately | Interface Endpoint + Direct Connect |
| Expose your service to other VPCs privately | PrivateLink (NLB + Interface Endpoint) |
| Block all S3 access outside company buckets | Endpoint policy on Gateway Endpoint |

> [!tip] Key Insight
> The ENI is a private door AWS punches through your VPC wall, connecting directly to their internal network. DNS returns the ENI's private IP → route table keeps it local → no internet involved.

> [!warning] Common Misconception
> Gateway Endpoints cannot be accessed from on-premises. If your on-premises servers need to reach S3 privately via Direct Connect, you need an Interface Endpoint for S3 (yes, S3 supports both).

## 32. Security Groups vs Network ACLs

### Security Groups (instance-level)
- **Stateful** — return traffic automatically allowed, no explicit outbound rule needed
- **Allow rules only** — no deny option
- All rules evaluated simultaneously — most permissive match wins
- Default: all inbound blocked, all outbound allowed
- Can reference other SGs instead of IP ranges (auto-updates as instances scale)

**SG-to-SG referencing (3-tier pattern):**
```
DB SG inbound: Allow port 3306 from sg-appserver-id
App SG inbound: Allow port 8080 from sg-webserver-id
```
No hardcoded IPs. Only the correct tier can reach each layer.

### Network ACLs (subnet-level)
- **Stateless** — must explicitly allow BOTH inbound AND outbound
- **Allow + Deny rules** — only control that can explicitly deny
- Rules processed in ORDER — lowest number first, first match wins, stops checking
- Default NACL: allows all. Custom NACL: denies everything until rules added.

**Ephemeral ports trap:**
Client connects on port 80 → response goes back on random port 1024-65535.
NACL must allow outbound TCP 1024-65535 or responses are silently blocked.

**Rule ordering matters:**
```
Rule 100: DENY  192.168.1.0/24  port 443   ← evaluated first
Rule 200: ALLOW 0.0.0.0/0       port 443   ← everyone else
```
Put DENY before the broader ALLOW or it never triggers.

### Comparison
| | Security Group | NACL |
|---|---|---|
| Level | Instance (ENI) | Subnet |
| State | Stateful | Stateless |
| Rules | Allow only | Allow + Deny |
| Evaluation | All rules, most permissive | Ordered, first match wins |
| Ephemeral ports | Not needed | Required (1024-65535) |

### Decision rule
- Block a specific IP → NACL (SGs cannot deny)
- Allow traffic between specific instances → SG-to-SG referencing
- Defense in depth → both (NACL at subnet + SG at instance)

> [!warning] Common Misconception
> NACL has no outbound rule → default deny blocks ALL responses even if inbound is allowed. Must explicitly allow outbound TCP 1024-65535 for response traffic.

## 33. VPC Flow Logs, Traffic Mirroring, Peering & Segmentation

### VPC Flow Logs
Captures network traffic metadata (not packet contents) at VPC / subnet / ENI level.
Destinations: CloudWatch Logs or S3.

**Key log fields:**
- `srcaddr` / `dstaddr` — source and destination IP
- `srcport` / `dstport` — ports
- `protocol` — 6=TCP, 17=UDP, 1=ICMP
- `action` — ACCEPT or REJECT (tells you if SG/NACL blocked it)

**NOT captured:** EC2 metadata traffic (169.254.169.254), DNS to Route 53 resolver, DHCP, actual packet contents.

**Query with Athena (S3 destination):**
```sql
SELECT srcaddr, dstaddr, dstport, action, count(*) as attempts
FROM vpc_flow_logs WHERE action = 'REJECT'
GROUP BY srcaddr, dstaddr, dstport, action ORDER BY attempts DESC;
```

**Transit Gateway Flow Logs** — same concept for cross-VPC traffic through TGW hub.

### VPC Traffic Mirroring
Copies actual **packet contents** (not just metadata) from a source ENI → NLB → security appliance (IDS/IPS).

```
EC2 ENI → mirrored copy → NLB → Suricata/Zeek/NGFW (full packet inspection)
```

Exam trigger: "inspect actual packet contents for malware/exfiltration" → Traffic Mirroring (not Flow Logs).

### VPC Peering
Direct VPC-to-VPC private communication. No internet. Cross-account and cross-region supported.

**Hard constraints:**
- **No transitive routing** — A↔B, B↔C does NOT mean A can reach C through B
- **No overlapping CIDR blocks** — peering fails if IP ranges conflict
- Route table entries + SG rules required on both sides

**The isolation pattern:**
```
Prod ←──peering──→ Shared-Services ←──peering──→ Dev
Prod and Dev have no direct peering → cannot reach each other
(non-transitive — traffic cannot flow A→B→C)
```

**Transit Gateway** = solves transitive routing. Regional hub — all VPCs connect to TGW, any can reach any other through it. Replaces N*(N-1)/2 peering connections.

### North/South vs East/West
- **North/South** — VPC ↔ internet or on-premises. Controlled at boundary (IGW, NAT, VPN).
- **East/West** — service-to-service within VPC or between VPCs. Controlled with SGs, NACLs, Network Firewall.

> [!tip] Key Insight
> VPC Peering is non-transitive by design. For hub-and-spoke where all VPCs need to reach each other, use Transit Gateway instead of a mesh of peering connections.

## 34. AWS Network Firewall, WAF & Firewall Manager

### The Firewall Family — Clear Separation
| Service | What it is | Layer | Scope |
|---|---|---|---|
| Security Groups | Per-instance virtual firewall | L3/L4 | One ENI |
| Network ACLs | Per-subnet virtual firewall | L3/L4 | One subnet |
| AWS Network Firewall | Managed IDS/IPS engine | L3-L7 | One VPC |
| AWS WAF | HTTP/HTTPS rule engine | L7 only | CloudFront, ALB, API GW |
| AWS Firewall Manager | Central management console | N/A | Manages all the above |

**Mental model:**
- WAF = bouncer at the website door (HTTP/HTTPS only)
- Network Firewall = checkpoint at VPC border (all traffic)
- Firewall Manager = security director managing all checkpoints across 50 accounts

### AWS Network Firewall — Three Rule Types

**1. Stateless rules (ID checker)**
- Checks each packet independently, no memory
- Matches: src IP, dst IP, src port, dst port, protocol
- First match wins, fast but dumb

**2. Stateful rules (detective)**
Three sub-formats:
- **Standard** — 5-tuple + connection state awareness
- **Suricata rules** — deep packet inspection, look inside payload:
  ```
  alert http any any -> any any (msg:"SQLi"; content:"UNION SELECT"; http.uri; sid:1001;)
  ```
- **Domain list rules** — inspect DNS queries and TLS SNI

**3. Domain list rules (phone tap)**
- Intercepts DNS queries before connection is made
- Block list: `*.c2-botnet.ru` → EC2 asks for IP → query blocked → connection never happens
- Works even when IP behind domain rotates constantly

```
Packet flow:
Stateless (IP/port check) → Stateful (behavior/payload) → Domain (DNS intercept) → PASS
```

### Network Firewall Architecture
Deployed in dedicated firewall subnets (one per AZ). Route tables force all VPC traffic through firewall endpoints.

### AWS WAF
HTTP/HTTPS only. Attaches to CloudFront, ALB, API Gateway, AppSync.
Blocks: SQLi, XSS, geo, IP, rate limits, bot traffic (managed rules).
Cannot inspect non-HTTP protocols.

### AWS Firewall Manager
NOT a firewall — a centralized deployment and enforcement tool.
Manages WAF, Network Firewall, Shield Advanced, Security Groups, DNS Firewall across all org accounts.
Requires AWS Organizations.
Exam trigger: "enforce consistent firewall rules across 20 accounts" → Firewall Manager.

### Decision Matrix
| Need | Tool |
|---|---|
| Block SQLi on web app | WAF |
| Block malicious domain C2 traffic | Network Firewall (domain list) |
| Deep packet inspection (Suricata) | Network Firewall |
| Org-wide consistent rules | Firewall Manager |
| Rate-limit API requests | WAF |
| Inspect SSH/FTP/non-HTTP traffic | Network Firewall |

> [!warning] Common Misconception
> Firewall Manager is not a firewall — it manages and deploys other firewalls centrally. WAF and Network Firewall are separate products for separate layers; they complement each other.

## 35. Hybrid Connectivity & Edge Security

### Site-to-Site VPN vs Direct Connect
| | Site-to-Site VPN | Direct Connect |
|---|---|---|
| Medium | IPsec tunnel over public internet | Dedicated private fiber |
| Setup | Minutes | Weeks/months |
| Bandwidth | Up to 1.25 Gbps | 1-100 Gbps |
| Latency | Variable | Consistent |
| Encrypted by default | Yes (IPsec) | **No** — must add VPN or MACsec |
| Cost | Low | Higher (cheaper at volume) |

**Direct Connect encryption trap:** Private ≠ encrypted. Add IPsec VPN tunnel over DX, or enable MACsec for wire-level encryption.

### AWS Transit Gateway
Regional hub connecting VPCs + VPN + Direct Connect.
- Solves transitive routing limitation of VPC Peering
- TGW route tables segment which attachments reach which networks
- Cross-region TGW peering for multi-region connectivity

### Client VPN vs Verified Access
| | Client VPN | Verified Access |
|---|---|---|
| Type | OpenVPN tunnel | Zero-trust app proxy |
| Auth | Certificate / AD / SAML | IAM Identity Center / OIDC |
| Device check | No | Yes |
| Access scope | Full network | Per-application |

### CloudFront Security Features
**OAC (Origin Access Control)** — restrict S3 to CloudFront only:
```json
{
  "Effect": "Allow",
  "Principal": { "Service": "cloudfront.amazonaws.com" },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*",
  "Condition": {
    "StringEquals": { "AWS:SourceArn": "arn:aws:cloudfront::123456789:distribution/XXXXX" }
  }
}
```
OAC = current. OAI = legacy (no SSE-KMS support, limited regions).

**Other CloudFront security features:**
- **Geo-restriction** — block/allow by country at edge
- **Signed URLs** — single file, time-limited access (CloudFront key pairs, not IAM)
- **Signed Cookies** — multiple files (streaming, course content)
- **Field-level encryption** — encrypt specific fields (e.g. credit cards) at edge with public key; only origin can decrypt
- **Custom headers** — secret header CloudFront adds; origin rejects requests without it (forces traffic through CF)
- **TLS minimum version** — enforce TLS 1.2+ at distribution level

> [!warning] Common Misconception
> Direct Connect is private but NOT encrypted by default. Compliance requiring encryption on the wire needs either IPsec VPN over DX or MACsec.

> [!tip] Key Insight

## 37. Compute Security

### IMDSv1 vs IMDSv2

## 44. Incident Response

### IR Lifecycle (6 phases)
Preparation → Detection → Containment → Eradication → Recovery → Post-Incident

### EC2 Compromise Response (heavily tested)

**Precise sequence:**
1. **Replace** SG with quarantine-sg (deny all in/out) — replace, not modify (avoid affecting other instances)
2. Attach deny-all IAM policy to instance role (stops AWS API calls from malware)
3. Snapshot EBS volume (forensic disk evidence)
4. Memory dump via SSM Run Command (volatile evidence — RAM not in EBS snapshot)
5. Tag instance as compromised (prevent accidental termination)
6. Analyze disk in **forensic VPC** (isolated, no internet, mount as volume not boot)
7. Terminate + redeploy from clean AMI
8. Rotate all credentials the instance had access to

**Hard rule: NEVER SSH into compromised instance** — contaminates evidence, risks credential theft.

**Automated pattern:**
```
GuardDuty → EventBridge → Lambda:
  1. Replace SG with quarantine-sg
  2. Attach deny-all to IAM role
  3. EBS snapshot (tagged forensic-evidence)
  4. SNS → security team
  5. SSM OpsItem for tracking
```

### Credential Compromise Response

**IAM user access key compromised:**
1. Inactivate the key immediately (`aws iam update-access-key --status Inactive`)
2. Investigate CloudTrail for all API calls made with the key
3. Delete the key
4. Create new key if needed
5. Remediate any damage done (delete unauthorized resources, revoke unauthorized policies)

**IAM role session (STS token) compromised:**
1. Attach inline deny-all policy to the role — covers all active sessions
2. OR use `aws:TokenIssueTime` condition to revoke sessions issued before the compromise time
3. Investigate via CloudTrail + Detective

**Root user compromised:**
1. Change root password immediately
2. Reset root MFA
3. Audit all IAM users/roles created or modified — attacker may have created backdoors
4. Check for unauthorized CloudTrail disabling, billing changes

### Automated Response Patterns

**Step Functions** — multi-step orchestrated response for complex workflows:
```
GuardDuty finding → EventBridge → Step Functions state machine:
  State 1: Isolate EC2 (Lambda)
  State 2: Snapshot EBS (Lambda)
  State 3: Wait for snapshot complete
  State 4: Notify security team (SNS)
  State 5: Create investigation ticket
```

**Config Rules auto-remediation** — continuous compliance:
- Rule: `ec2-imdsv2-required` → FAILED → SSM doc forces IMDSv2
- Rule: `s3-bucket-public-read-prohibited` → FAILED → Lambda disables public access

**AWS Fault Injection Service (FIS)** — chaos engineering for testing IR plans. Inject failures (CPU stress, network latency, instance termination) to validate your runbooks work before a real incident.

### Investigation Tools
- **Detective** — graph visualization, entity relationships, timeline
- **CloudTrail** — who made what API call
- **VPC Flow Logs** — network-level evidence (what IPs communicated)
- **Athena** — SQL queries on CloudTrail/Flow Log data in S3
- **SSM OpsCenter** — operational item tracking for incidents

> [!tip] Key Insight
> Replace SG (not modify) during EC2 isolation — modifying a shared SG accidentally isolates other instances using it. Always use a dedicated quarantine SG.

> [!warning] Common Misconception
> EBS snapshots capture disk only — NOT memory. For forensic completeness, memory dump must be captured separately via SSM Run Command before the instance state changes.

## 46. Practice Scenario Review

### Scenario 1 — Cross-account S3 access with SCP
SCP deny condition was `StringNotEquals region us-east-1`. Bucket IS in us-east-1 → condition false → SCP passes.
Cross-account: identity policy (s3:*) AND bucket policy (s3:GetObject) both grant → **ALLOW**.
Trap: seeing "SCP with Deny" doesn't mean it always blocks. Evaluate the condition carefully.

### Scenario 2 — Root user compromise response
Order: (C) Change password + invalidate session FIRST → stops active attacker.
(A) is a trap — root should have no access keys. Rotating nonexistent keys wastes time.
Then: verify MFA, audit CloudTrail for backdoors (new IAM users, policy changes), check resource damage.

### Scenario 3 — FIPS 140-2 Level 3 + AWS cannot access key
Answer: **SSE-KMS with custom key store backed by CloudHSM**.
- Standard CMK = Level 2, AWS holds key material
- CloudHSM = Level 3, YOU own the HSM, AWS is proxy only
- Custom key store = looks like normal CMK to SDK, crypto happens in your CloudHSM
- "Even if AWS is compromised" → key safe because it lives in your hardware

### Scenario 4 — Private subnet KMS calls failing after NAT deleted
Answer: **Interface Endpoint for KMS**.
NAT works but costs money + unnecessary. Elastic IP = security violation. SSE-S3 changes encryption model.
Rule: "private subnet + AWS service + no internet + calls failing" → Interface Endpoint.

### Scenario 5 — Even AWS cannot read DynamoDB fields
Answer: **DynamoDB Encryption Client (client-side)**.
SSE options (CMK, AWS managed) = AWS performs crypto = AWS in trust chain.
Client-side = ciphertext arrives at DynamoDB = AWS never sees plaintext.
Rule: "even AWS cannot read it" → client-side encryption always.

### Scenario 6 — Prevent privilege escalation across 200 accounts
Answer: **SCP denying iam:CreateRole without approved boundary** (preventive, instant, all accounts).
Plus Config conformance pack (remediates existing roles already created without boundaries).
Rule: "prevent across entire org immediately" → SCP. "Remediate existing violations" → Config auto-remediation.

### Scenario 7 — L7 botnet attack rotating 3,000 IPs
Answer: **WAF Bot Control managed rule group + rate-based rule on /api/search**.
- Shield Advanced = L3/L4, DRT not instant
- Per-IP rate limiting = 3,000 IPs × 17 req/min each = all under threshold, useless
- Bot Control = behavioral signals (JS execution, user-agent fingerprint, timing) catches all IPs as one category
Rule: "distributed botnet rotating IPs, valid HTTP, CPU exhaustion" → WAF Bot Control.
